# RV64.md - Building a RISC-V 64-bit Operating System

A complete, build-oriented engineering reference for writing an RV64 kernel that boots in
**supervisor mode under OpenSBI on the QEMU `virt` machine** - the standard development target -
with the production-aware caveats you need to also run on real hardware.

**Vantage:** mid-2026. Facts here were verified against primary sources (the RISC-V
Unprivileged/Privileged ISA manuals, the ratified SBI specification, the OpenSBI source, the
Devicetree and VIRTIO specifications, QEMU's `hw/riscv/virt.c`) and against working code (the
`mit-pdos/xv6-riscv` `riscv` branch, which is the canonical teaching kernel and is cited
throughout). Where xv6 takes a simplification that is non-portable, the production alternative is
called out. Confidence is implicit unless a value is flagged as version-dependent or unverified.

**How to use this.** Parts 0-2 get you to first output. Parts 3-7 are the kernel substrate
(traps, time, paging, atomics, devices). Part 8 builds the process/SMP/concurrency layer. Part 9 is
optional modern features. Part 10 is targeting/landscape context. The appendices are quick-reference
tables and a dependency-ordered bringup checklist (Appendix G) - start there if you want the build
order.

A recurring theme: **most of these details are "correct on QEMU, broken on real hardware" traps**
(A/D bits, `fence.i`, missing `sfence.vma`, missing MMIO fences). They are flagged inline and
collected in Appendix F.

# Part 0 - Orientation and targeting decisions

## 0.1 The privilege model

RISC-V has three privilege levels, encoded as a 2-bit mode:

| Level | Enc | Role | Where your kernel fits |
|---|---|---|---|
| **Machine (M)** | `11` | Firmware. Unrestricted; owns physical memory, PMP, the real interrupt wiring. | OpenSBI lives here. You do not. |
| **Supervisor (S)** | `01` | OS kernel. Gets the MMU (`satp`) and a restricted trap apparatus. | **Your kernel runs here.** |
| **User (U)** | `00` | Applications. | Your processes run here. |

Two rules govern everything:

1. **Traps go to the same or a higher privilege**, never lower. M-mode can *delegate* S/U-originated
   traps down to S so the kernel handles them without an M round-trip (OpenSBI does this for you).
2. **`ecall` traps to the next level up.** `ecall` from U traps to S (your syscall path); `ecall`
   from S traps to M (your SBI calls to firmware).

The recommended model is the xv6/Linux split: **M = OpenSBI firmware, S = your kernel, U = your
processes.** You boot straight into S-mode; OpenSBI has already done the M-mode setup. You manage
only the S-level CSRs (`sstatus`, `sie`, `stvec`, `sepc`, `scause`, `satp`, ...); touching the M-level
CSRs from S-mode traps.

## 0.2 The target machine: QEMU `virt`

Develop against `qemu-system-riscv64 -machine virt`. It is a synthetic board with a stable memory
map (full table in Appendix A). The pieces you touch first:

| Device | Base | Note |
|---|---|---|
| Boot ROM (reset vector) | `0x00001000` | where the hart starts |
| sifive_test (poweroff) | `0x00100000` | write `0x5555` to exit QEMU cleanly |
| CLINT (timer/IPI) | `0x02000000` | M-mode timer + software interrupts |
| PLIC (external IRQs) | `0x0C000000` | device interrupt controller |
| NS16550A UART | `0x10000000` | the serial console |
| virtio-mmio slots (8) | `0x10001000` | `+0x1000` stride; disk/net |
| Main RAM (DRAM) | `0x80000000` | OpenSBI at base, **your kernel at `0x80200000`** |

QEMU **auto-generates a device tree** describing this map and passes a pointer to it in `a1` at boot
(Part 7). Change `-m`/`-smp` and the regenerated DTB reflects it - which is why you discover hardware
from the DTB instead of hardcoding.

## 0.3 The ISA baseline to target

- **`rv64gc`** (= `rv64imafdc` + Zicsr + Zifencei) with the **`lp64d`** ABI is the de-facto baseline
  every distro and board assumes, and what QEMU `virt` presents. Target this.
- **Profiles** formalize baselines: **RVA20** is essentially `rv64gc`; **RVA22** adds bit-manip and
  cache ops; **RVA23** (ratified 2024-10-17) makes **Vector (V) and the Hypervisor (H) extension
  mandatory** and is what Ubuntu/Android are moving to - but RVA23 silicon barely existed through
  2025 (Part 10). For a from-scratch kernel, build for `rv64gc`/RVA20 and feature-detect anything
  beyond it at runtime (the device tree's `riscv,isa-extensions`, or `riscv_hwprobe` on Linux).
- The **ABI** pairs with the ISA: `lp64d` (hard-float double) requires `d` in `-march`. A mismatch is
  a link error or a silently wrong calling convention.

## 0.4 The toolchain and build flags (C/Rust)

For a freestanding S-mode kernel linked at `0x80200000`:

| Flag | Why | Mandatory |
|---|---|---|
| `-march=rv64gc` | base + IMAFD + C; matches QEMU `virt` | yes (pick one, match `-mabi`) |
| `-mabi=lp64d` | hard-float double ABI; must pair with a `d`-containing `-march` | yes |
| **`-mcmodel=medany`** | PC-relative addressing reachable at `0x80200000`. `medlow` can only address a 2 GiB window around address 0 and **cannot reach `0x80200000`** - using it bricks the kernel. | **yes** |
| `-ffreestanding` | no hosted libc assumptions | yes |
| `-nostdlib` | you provide `_start`, no crt0/libc | yes |
| `-fno-stack-protector` | no `__stack_chk_guard` symbol exists | yes |
| `-fno-pic` / `-no-pie` | static, fixed-load, no GOT/PLT | yes |
| `-Wl,--no-relax` and/or `.option norelax` around the `gp` load | disable linker relaxation, especially **GP relaxation** | strongly recommended (see 1.3) |

Rust: `riscv64gc-unknown-none-elf` target, `panic=abort`, a `#![no_std] #![no_main]` crate, a custom
target/linker script. Go: the `embeddedgo` (`GOOS=noos`) toolchain lays out the image from an `-M
load:size` linker flag rather than a `.ld` script.

## 0.5 The boot model choice

- **S-mode payload under OpenSBI (recommended, this document's path):** `-bios default` gives you
  OpenSBI in M-mode; `-kernel your.elf` is loaded as an S-mode payload entered at `0x80200000`. You
  inherit a configured machine and talk to firmware over SBI. This is how Linux and most real boards
  boot.
- **M-mode kernel (`-bios none`, the classic xv6 setup):** your kernel *is* the firmware, runs in
  M-mode, links at `0x80000000`, and does its own M-mode setup before dropping to S. More to do,
  but no SBI dependency. xv6 does this; its `start.c` is the M-mode setup. **Note when reading xv6:
  its privilege model differs from the OpenSBI path** - use its layout/asm patterns, not its boot
  model, if you target OpenSBI.

# Part 1 - Bringup: power-on to your first instruction

## 1.1 The boot chain and the `a0`/`a1` handoff contract

```
power-on -> ROM/reset vector (M) -> [first-stage loader] -> OpenSBI (M, resident)
         -> your kernel (S) @ 0x80200000
```

OpenSBI configures PMP (to grant S/U access to RAM and MMIO while walling off its own region),
delegates the usual S/U traps, sets `mstatus.MPP = S`, `mepc` = your entry, and `mret`s into your
kernel. **The exact state you inherit on the boot hart** (verified from OpenSBI
`sbi_hart_switch_mode`):

| Register | Value at entry |
|---|---|
| `a0` | **hartid** of the boot hart |
| `a1` | **physical address of the device tree blob (DTB)** |
| privilege | **S-mode** |
| `sstatus.SIE` | **0** (interrupts globally disabled) |
| `sie` | **0** (no individual S-interrupt enabled) |
| `satp` | **0** (MMU off, bare physical addressing) |
| `sscratch` | 0 |
| `stvec` | = your entry address (you must overwrite it with your trap vector) |

This `a0`=hartid / `a1`=DTB convention is universal across RV64 S-mode kernels (it is also the Linux
boot contract, which additionally specifies `satp = 0`).

**Only the boot hart enters your kernel.** Under modern OpenSBI, secondary harts are parked in
firmware in the STOPPED state, awaiting an SBI HSM `hart_start` call (Part 8.4). The old "all harts
enter and spin on hartid" model (classic xv6) is not how ordered boot works.

What firmware does **not** do for you: zero your `.bss`, give you a usable stack, or set any register
beyond `a0`/`a1`. Your `_start` does that.

## 1.2 The linker script

A correct minimal `kernel.ld` for the S-mode-under-OpenSBI path:

```ld
OUTPUT_ARCH("riscv")
ENTRY(_start)                       /* e_entry; QEMU's ELF loader jumps here */

SECTIONS
{
  . = 0x80200000;                   /* OpenSBI next-stage address on QEMU virt
                                       (NOT 0x80000000, which OpenSBI occupies) */
  .text : {
    *(.text._start)                 /* _start FIRST - lowest address */
    *(.text .text.*)
  }
  . = ALIGN(4096);
  .rodata : { *(.srodata .srodata.*) *(.rodata .rodata.*) }

  . = ALIGN(4096);
  .data : {
    PROVIDE(__global_pointer$ = . + 0x800); /* anchor at the START of small-data so the */
    *(.sdata .sdata.*) *(.data .data.*)     /* GP-relative +-2KiB window covers .sdata.   */
  }                                         /* omit gp entirely if you build --no-relax.  */
  . = ALIGN(4096);
  PROVIDE(__bss_start = .);
  .bss : { *(.sbss .sbss.*) *(.bss .bss.*) *(COMMON) }
  PROVIDE(__bss_end = .);

  . = ALIGN(16);
  PROVIDE(_end = .);

  . = ALIGN(16);                    /* boot stack, 16 KiB, 16-byte aligned */
  PROVIDE(_stack_bottom = .);
  . += 16 * 1024;
  PROVIDE(_stack_top = .);
}
```

Key points:

- **`. = 0x80200000`**, not `0x80000000`. Linking at `0x80000000` under `-bios default` collides with
  OpenSBI (`rom: requested regions overlap`).
- **`_start` is at the lowest address.** A flat binary is entered at offset 0 = the load address; if
  `_start` is not first, you jump into the wrong code. (Even with an ELF, where the entry comes from
  `e_entry`, keep `_start` first - it matches the flat fallback and is the universal convention.)
- Symbols the stub needs: `__bss_start`/`__bss_end` (BSS-clear bounds), `_end` (heap base),
  `_stack_top` (initial `sp`), and `__global_pointer$` (only if you set `gp`).
- Page-align `.text`/`.rodata`/`.data`/`.bss` so you can later map them with distinct R/W/X
  permissions once paging is on (Part 5). `__global_pointer$` is anchored at the start of small-data
  (`.sdata`, +0x800) so the GP-relative +-2 KiB window covers it; this matters only if you keep linker
  relaxation on (with `--no-relax` you can drop `gp` and this symbol entirely).
- When you add user mode (Part 3.3 / Part 8) you will also reserve a **page-aligned trampoline
  section** here - its own output section holding `uservec`/`userret`, with `ALIGN(4096)` on both sides
  and an `ASSERT` that it fits in one 4 KiB page - so it can be mapped at the same VA in the kernel and
  in every user page table. The boot script above omits it because the minimal S-only kernel (Part 3.2)
  does not need it.

## 1.3 The `_start` entry stub

```asm
.section .text._start
.globl _start
_start:
    # 1. set gp under norelax (omit entirely if you build with --no-relax)
    .option push
    .option norelax
    la   gp, __global_pointer$      # MUST be norelax, or the linker relaxes this
    .option pop                     #   to `mv gp, gp` and gp stays garbage

    # 2. park secondary harts (belt-and-suspenders; ordered boot sends only hart 0)
    bnez a0, park                   # a0 = hartid

    # 3. stack pointer, 16-byte aligned (ABI requirement at any call boundary)
    la   sp, _stack_top
    andi sp, sp, ~15

    # 4. enable FP state so lp64d FP instructions don't trap (see note)
    li   t0, (1 << 13)              # sstatus.FS = 0b01 (Initial); FS field is bits [14:13]
    csrs sstatus, t0

    # 5. preserve a0/a1 across BSS clear (memset clobbers caller-saved a0/a1)
    mv   s0, a0                     # s0/s1 are callee-saved; survive the loop
    mv   s1, a1
    la   t0, __bss_start
    la   t1, __bss_end
1:  bgeu t0, t1, 2f
    sd   zero, 0(t0)                # firmware does NOT zero BSS
    addi t0, t0, 8
    j    1b
2:  mv   a0, s0                     # restore hartid
    mv   a1, s1                     # restore DTB pointer

    # 6. into high-level code (never returns)
    call kmain                      # kmain(hartid = a0, dtb = a1)
3:  wfi
    j    3b
park:
    wfi
    j    park
```

The non-obvious correctness points, each of which is a real "won't boot" bug:

- **GP relaxation:** with relaxation on, the linker rewrites symbol loads near `__global_pointer$`
  into `addi rd, gp, off`, assuming `gp` is already set. In `_start`, `gp` is garbage until you load
  it - and the load itself can get relaxed to `mv gp, gp`. Wrap it in `.option norelax` and/or build
  with `-Wl,--no-relax`. (Or skip `gp` entirely and always build `--no-relax`, as Linux does.)
- **Preserve `a0`/`a1`:** they hold hartid and the DTB pointer and are caller-saved temporaries; a
  `memset`/BSS loop clobbers them. Stash them in callee-saved registers (`s0`/`s1`) first. This is a
  classic boot bug.
- **Clear BSS:** neither OpenSBI nor QEMU zeros it; uninitialized globals hold garbage.
- **16-byte `sp` alignment:** required at any `call`; an unaligned `sp` misaligns spills and faults.
- **Enable `sstatus.FS`:** with `lp64d` the compiler emits FP instructions; the reset `FS` is Off, so
  the first FP op raises an illegal-instruction trap. Set `FS = Initial` (`0b01`, bit 13) before any
  FP runs. (Vector is the analogous `VS` field, bits [10:9] - leave Off unless you target V.)

## 1.4 Loading the image and running QEMU

- **Pass the ELF directly to `-kernel`.** QEMU parses `PT_LOAD` segments and uses `e_entry`; OpenSBI
  (`fw_dynamic`, the QEMU default) receives the entry/load and `mret`s to it in S-mode. **A flat
  binary from `objcopy -O binary` is NOT loaded by `-kernel` on `qemu-system-riscv64`** - that is a
  silent no-boot trap. (A raw binary loads via `-bios`, which replaces OpenSBI.) Use the ELF.

Dev / interactive:
```sh
qemu-system-riscv64 -machine virt -m 128 -smp 1 -bios default -nographic -kernel kernel.elf
# quit with Ctrl-A then x
```

CI / non-interactive (so piped stdin works and a crash exits instead of looping):
```sh
qemu-system-riscv64 -machine virt -m 128 -smp 1 -bios default \
  -display none -serial stdio -no-reboot -kernel kernel.elf
```

GDB:
```sh
qemu-system-riscv64 -machine virt -m 128 -smp 1 -bios default -nographic -kernel kernel.elf -s -S
#   -s = -gdb tcp::1234 ;  -S = freeze at startup
riscv64-elf-gdb kernel.elf -ex 'target remote :1234'
#   then: info registers satp sstatus scause sepc stval ; x/8i $pc
```

Make QEMU exit from inside the kernel via the **sifive_test** device at `0x100000`: a 32-bit store of
`0x5555` = `FINISHER_PASS` (clean poweroff), `0x3333` = fail, `0x7777` = reset.

## 1.5 First output

Before you have a UART driver, print via the SBI console (Part 2): legacy `console_putchar`
(EID `0x01`) is one ecall per byte and works everywhere. This is the fastest path to a "hello" and to
verifying you reached S-mode.

# Part 2 - SBI: the firmware ABI

SBI (Supervisor Binary Interface) is how your S-mode kernel asks the M-mode firmware (OpenSBI) for
services. You issue an `ecall` (which traps S->M); OpenSBI services it and returns.

## 2.1 The ecall ABI

| Register | Role |
|---|---|
| `a7` | **Extension ID (EID)** |
| `a6` | **Function ID (FID)** (for v0.2+ extensions) |
| `a0`-`a5` | arguments |
| `a0` (return) | **error code** |
| `a1` (return) | **return value** |

All registers except `a0`/`a1` are preserved across the call. Modern EIDs spell their name in ASCII
(e.g. "TIME" = `0x54494D45`). The **legacy** (v0.1) calls (EIDs `0x00`-`0x0F`) use a weaker
convention: the EID is the function, there is no FID, and only `a0` is returned/preserved.

Standard error codes: `SBI_SUCCESS`=0, `ERR_FAILED`=-1, `ERR_NOT_SUPPORTED`=-2, `ERR_INVALID_PARAM`=-3,
`ERR_DENIED`=-4, `ERR_INVALID_ADDRESS`=-5, `ERR_ALREADY_AVAILABLE`=-6, ... through `ERR_DENIED_LOCKED`=-14.

A register-correct wrapper (C / GCC inline asm):

```c
struct sbiret { long error; long value; };
static struct sbiret sbicall(long eid, long fid,
        unsigned long a0,unsigned long a1,unsigned long a2,
        unsigned long a3,unsigned long a4,unsigned long a5) {
    register unsigned long r0 asm("a0")=a0, r1 asm("a1")=a1, r2 asm("a2")=a2;
    register unsigned long r3 asm("a3")=a3, r4 asm("a4")=a4, r5 asm("a5")=a5;
    register long r6 asm("a6")=fid;          /* omit for legacy calls */
    register long r7 asm("a7")=eid;
    asm volatile("ecall" : "+r"(r0),"+r"(r1)
                 : "r"(r2),"r"(r3),"r"(r4),"r"(r5),"r"(r6),"r"(r7) : "memory");
    return (struct sbiret){ .error=(long)r0, .value=(long)r1 };
}
```

## 2.2 The calls a kernel actually needs

| Call | EID (`a7`) | FID (`a6`) | Args | Returns |
|---|---|---|---|---|
| Base: get_spec_version | `0x10` | 0 | - | minor[23:0], major[30:24] |
| Base: probe_extension | `0x10` | 3 | a0=EID | 0=absent, nonzero=present |
| Base: get_impl_id | `0x10` | 1 | - | 1 = OpenSBI |
| TIME: set_timer | `0x54494D45` | 0 | a0=absolute `time` value | arms next timer; clears pending STIP |
| IPI: send_ipi | `0x735049` | 0 | a0=hart_mask, a1=hart_mask_base | raises SSIP on targets |
| HSM: hart_start | `0x48534D` | 0 | a0=hartid, a1=start_addr(phys), a2=opaque | starts a hart in S-mode |
| HSM: hart_get_status | `0x48534D` | 2 | a0=hartid | state 0..6 |
| RFENCE: remote_sfence_vma | `0x52464E43` | 1 | a0=mask,a1=base,a2=start,a3=size | remote TLB shootdown |
| RFENCE: remote_fence_i | `0x52464E43` | 0 | a0=mask, a1=base | remote `fence.i` |
| SRST: system_reset | `0x53525354` | 0 | a0=type (0=shutdown,1=cold,2=warm), a1=reason | poweroff/reboot |
| DBCN: console_write | `0x4442434E` | 0 | a0=len, a1=addr_lo, a2=addr_hi (**physical**) | bytes written |
| DBCN: console_write_byte | `0x4442434E` | 2 | a0=byte | blocking single byte |
| Legacy: console_putchar | `0x01` | - | a0=byte | (legacy; simplest early console) |
| Legacy: console_getchar | `0x02` | - | - | byte, or -1 if none |

Console choice: legacy `putchar`/`getchar` (one byte per ecall) is the simplest bring-up console.
**DBCN** (`console_write` with a physical buffer address) is the modern, batched path - feature-detect
it with `probe_extension(0x4442434E)` and fall back to legacy. While paging is off (or identity
mapped), a virtual buffer address equals its physical address; after paging, translate before the
call.

HSM-started secondary harts enter at `start_addr` in **S-mode, `satp`=0, `sstatus.SIE`=0**, with
`a0`=hartid and `a1`=the `opaque` value you passed.

Ratified SBI versions (verified from the spec repo's git tags): **v2.0 = 2024-02-01**, **v3.0 =
2025-07-16** (these are routinely misquoted - the older "2023-10-06" is a freeze, not ratification).
OpenSBI master implements v3.0.

## 2.3 What OpenSBI delegates (so you know which traps you own)

OpenSBI's generic platform (QEMU `virt`) sets `medeleg`/`mideleg` so most S/U traps come straight to
your S-mode handler. Verified from OpenSBI `delegate_traps()`:

| Trap | Cause | Delegated to S? | Handled by |
|---|---|---|---|
| Instruction address misaligned | 0 | **yes** | your kernel |
| Breakpoint | 3 | **yes** | your kernel |
| **Ecall from U-mode** | 8 | **yes** | your kernel (the syscall path) |
| Instruction page fault | 12 | **yes** | your kernel |
| Load page fault | 13 | **yes** | your kernel |
| Store/AMO page fault | 15 | **yes** | your kernel |
| Software check | 18 | **yes** | your kernel |
| S-software / S-timer / S-external interrupts | 1 / 5 / 9 | **yes** | your kernel |
| Illegal instruction | 2 | no | M-mode (CSR emulation) |
| Misaligned load / store | 4 / 6 | no | M-mode (emulation) |
| Access faults (instr/load/store) | 1 / 5 / 7 | no | M-mode (may reflect back to S) |
| **Ecall from S-mode** | 9 | **no (kept in M)** | OpenSBI - this is how it receives your SBI calls |

So your kernel owns: misaligned-fetch, breakpoint, U-ecall, all three page faults, software-check, and
the three S-level interrupts. Do **not** try to delegate ecall-from-S to yourself - you would loop
your own SBI calls back into your trap handler.

If you instead take the **M-mode boot path** (Part 0.5, `-bios none`) and write `medeleg`/`mideleg`
yourself, note they are **WARL**: a naive "delegate everything" write (`0xffff`) does not all stick.
On QEMU `virt` only `MIDELEG = 0x222` (SSIP/STIP/SEIP) and `MEDELEG = 0xb109` (misaligned-fetch,
breakpoint, ecall-from-U, the three page faults) take effect - the exact mask is QEMU-version-dependent.
The machine-level timer/software/external interrupts **cannot** be delegated to S, which is why an
M-mode kernel needs its own timer relay (or the Sstc `stimecmp` path). Under `-bios default` this is
moot - OpenSBI did the delegation already.

# Part 3 - Traps and interrupts

Every syscall, fault, and interrupt enters the kernel through one door: a trap to `stvec`.

## 3.1 The trap model

`scause` top bit (bit 63 on RV64) = **1 for an interrupt, 0 for an exception**. The low bits are the
cause code. Compare the whole 64-bit value against constants (e.g. `0x8000000000000005` = S-timer).

Exception causes: `0` instr misaligned, `1` instr access fault, `2` illegal instruction, `3`
breakpoint, `4` load misaligned, `5` load access fault, `6` store/AMO misaligned, `7` store/AMO access
fault, `8` ecall-from-U, `9` ecall-from-S, `11` ecall-from-M, `12` instr page fault, `13` load page
fault, `15` store/AMO page fault. (Newer spec adds `16` double-trap, `18` software-check, `19`
hardware-error.)

Interrupt causes (bit 63 set): `1` S-software, `5` S-timer, `9` S-external (and `3`/`7`/`11` are the
M-mode variants; `13` is counter-overflow with Sscofpmf). **The cause number equals the bit position
in `sip`/`sie`.**

**What the hardware does on a trap** (atomically, before your handler's first instruction):
`sstatus.SPIE <- SIE`; `SIE <- 0` (interrupts off in the handler); `SPP <-` the mode you came from (0=U,
1=S); `sepc <-` faulting/interrupted PC; `scause <-` cause; `stval <-` faulting address or instruction;
privilege <- S; `pc <- stvec`. It does **not** save any GPRs, switch stacks, or change `satp` - that
is all software's job.

**`sret`** reverses the state machine: privilege <- `SPP`; `SIE <- SPIE`; `SPIE <- 1`; `SPP <- U`;
`pc <- sepc`. So to return to user you set `SPP=0`; to stay in the kernel you leave `SPP=1`.

**`stvec`** holds `BASE | MODE` in the low 2 bits: `MODE=0` Direct (all traps -> BASE), `MODE=1`
Vectored (interrupts -> BASE + 4*cause). BASE must be >=4-byte aligned. Use Direct mode and dispatch
in software (what xv6 does) - keep your handler 4-byte aligned so the MODE bits stay 0.

## 3.2 The minimal S-only handler

If you have no user mode yet (everything runs in S), one Direct-mode handler suffices: there is no
`satp` switch, no trampoline, no fixed-VA trapframe. Install it once per hart (`csrw stvec, kvec`) and
save to the current stack. Exact pattern (xv6 `kernelvec`):

```asm
.align 4
kernelvec:
    addi sp, sp, -256            # 256-byte frame (16-byte aligned)
    sd ra,0(sp); sd gp,16(sp); sd t0,32(sp); sd t1,40(sp); sd t2,48(sp)
    sd a0,72(sp); sd a1,80(sp); sd a2,88(sp); sd a3,96(sp); sd a4,104(sp)
    sd a5,112(sp); sd a6,120(sp); sd a7,128(sp)
    sd t3,216(sp); sd t4,224(sp); sd t5,232(sp); sd t6,240(sp)
    call kerneltrap              # C handler
    ld ra,0(sp); ld gp,16(sp); ld t0,32(sp); ...   # restore the same set
    addi sp, sp, 256
    sret
```

Save the **caller-saved** set (`ra`, `t0-t6`, `a0-a7`); the C handler preserves `s0-s11`/`sp` itself.
Save to the *interrupted thread's own* stack (not a shared scratch) so the saved state survives any
context switches the handler triggers. Do **not** save/restore `tp` here - it holds the hartid, and a
`yield()` inside the handler may migrate you to another CPU; reloading a stale `tp` corrupts per-CPU
state.

## 3.3 The trampoline and trapframe (for user mode)

When a U-mode trap fires, you enter S-mode but **`satp` still points at the user page table**. To
reach the kernel you `csrw satp, kernel_satp` - but the instruction right after that is fetched
through the *new* table. The only way it stays mapped is if the switching code lives at the **same
virtual address in both the user and kernel page tables**. That shared page is the **trampoline**.

Map it (no `PTE_U`) at a fixed high VA in the kernel table and every user table; map a per-process
**trapframe** page just below it (R/W, no U). Pre-load `sscratch` with the trapframe VA before
entering user mode.

The trapframe layout (xv6, **288 bytes total**): the first 5 fields are kernel-supplied bootstrap
values, then all 31 user GPRs:

```c
struct trapframe {
  /*  0 */ uint64 kernel_satp;    /*  8 */ uint64 kernel_sp;
  /* 16 */ uint64 kernel_trap;    /* 24 */ uint64 epc;        /* 32 */ uint64 kernel_hartid;
  /* 40 */ ra; sp; gp; tp; t0; t1; t2; s0; s1;
  /* 112 */ a0; a1; ... a7; s2; ... s11; t3; t4; t5; /* 280 */ t6;
};
```

`uservec` (in the trampoline, user table still active):

```asm
uservec:
    csrrw a0, sscratch, a0       # swap: a0 <- trapframe VA, sscratch <- user a0
    sd ra,40(a0); sd sp,48(a0); sd gp,56(a0); sd tp,64(a0)   # save 30 GPRs ...
    sd t0,72(a0); ... ; sd t6,280(a0)
    csrr t0, sscratch; sd t0,112(a0)   # recover & store user a0
    ld sp,  8(a0)                # kernel_sp
    ld tp, 32(a0)                # kernel_hartid
    ld t0, 16(a0)                # &usertrap
    ld t1,  0(a0)                # kernel_satp
    sfence.vma zero, zero        # finish loads against the USER table first
    csrw satp, t1                # switch to kernel table
    sfence.vma zero, zero        # flush stale user TLB entries
    jr t0                        # -> usertrap() (C, kernel table, kernel stack)
```

All five trapframe loads happen **before** `csrw satp`, because afterward the trapframe VA is no
longer mapped. The `sfence.vma; csrw satp; sfence.vma` bracket is mandatory in both directions.

`userret` (return path) does the reverse: `sfence.vma; csrw satp, user_satp; sfence.vma`, restore all
GPRs from the trapframe (a0 last, since it is the base pointer), `sret`. Before calling it, the C code
sets `sstatus.SPP=0` and `SPIE=1`, loads `sepc` from the trapframe's `epc`, and refills the kernel
bootstrap fields for the next entry.

## 3.4 The C dispatch and its gotchas

```c
uint64 usertrap(void) {
    w_stvec((uint64)kernelvec);              // repoint to kernel handler IMMEDIATELY
    struct proc *p = myproc();
    p->trapframe->epc = r_sepc();            // save user pc before re-enabling interrupts
    uint64 cause = r_scause();
    if (cause == 8) {                        // ecall from U -> syscall
        p->trapframe->epc += 4;              // step past the ecall (always 4 bytes - see below)
        intr_on();                           // safe ONLY now (epc/scause already saved)
        syscall();                           // number in a7, args a0-a5, result -> a0
    } else if (devintr()) {                  // interrupt: timer / external / software
        // handled
    } else if ((cause==13 || cause==15) && handle_page_fault(r_stval(), cause)) {
        // demand paging / COW resolved at the faulting VA (stval)
    } else {
        // cause 12 (instr fault), 2 (illegal), ... -> kill the process
    }
    if (timer_fired) yield();                // preempt
    return prepare_return();                 // sets SPP/SPIE/sepc, returns user satp
}
```

Correctness gotchas (these cause silent, intermittent corruption):

- **Nested traps:** the S-mode trap CSRs (`sepc`/`scause`/`sstatus`/`stval`) are a **one-deep stack**.
  A second trap overwrites them. So you must spill `sepc` (and anything you need) to memory **before**
  `intr_on()`. Run kernel-context traps entirely with interrupts off.
- **`sepc += 4`:** on an `ecall`, `sepc` points *at* the `ecall`; without advancing it you re-execute
  the ecall forever. `+4` is always correct for syscalls because **there is no compressed `c.ecall`** -
  `ecall` is always a 32-bit instruction. (For *other* emulated instructions you must decode width
  from the opcode's low bits: `(insn & 3) != 3` => 2 bytes.)
- **`stvec` window:** repoint `stvec` from `uservec` to `kernelvec` as the first thing in `usertrap`,
  and back to `uservec` (interrupts off) just before `sret`. A trap taken while `stvec` points the
  wrong way for the current privilege is fatal.
- **`tp`:** user code can clobber it; `uservec` reloads it from `kernel_hartid`. In the kernel, `tp` is
  the hartid - handle as in 3.2.
- **FP/vector state:** if user code uses FP (a managed-runtime kernel will), you must save/restore
  `f0-f31`+`fcsr` on the trap path (gated by `sstatus.FS == Dirty`) or you silently corrupt user FP
  registers. xv6's user code uses no FP, so its trapframe has no FP slots - do not copy that omission
  if your user code uses FP.

# Part 4 - Interrupts and time

Two controllers, two jobs. **CLINT** = timer + software interrupts (per-hart). **PLIC** = external
device interrupts.

## 4.1 CLINT and the timer

QEMU `virt` CLINT at `0x02000000`:

| Register | Address | Effect |
|---|---|---|
| `msip[hart]` | `base + 4*hart` | write bit 0 -> M-software interrupt (IPI) |
| `mtimecmp[hart]` | `base + 0x4000 + 8*hart` | timer fires when `mtime >= mtimecmp` |
| `mtime` | `base + 0xBFF8` | free-running counter |

The timebase is **10 MHz** by default on QEMU `virt` (read `/cpus/timebase-frequency` from the DTB for
real boards). But `mtimecmp` is an **M-mode** register - an S-mode kernel cannot write it directly. Two
ways to set a timer:

1. **SBI `set_timer`** (TIME extension): portable, one ecall into firmware per tick.
2. **Sstc extension - `stimecmp` (CSR `0x14D`):** a supervisor timer interrupt becomes pending when
   `time >= stimecmp`, written **directly from S-mode**, no ecall. This requires M-mode to have set
   `menvcfg.STCE` (OpenSBI sets it **only if it detects Sstc in hardware**) and `mcounteren.TM` (which
   OpenSBI sets). If Sstc is absent, writing `stimecmp` traps as illegal-instruction.

**Decision: detect, do not assume.** Probe Sstc via the DTB ISA string (or attempt and trap-handle),
and fall back to SBI `set_timer`. QEMU `virt` usually exposes Sstc on recent versions, but this is
version-dependent. Reading the `time` CSR (`0xC01`) from S-mode is always allowed (OpenSBI sets
`mcounteren.TM`).

To schedule the next tick: `set_timer(time_now + interval)` (or `csrw stimecmp, time_now + interval`).
The same future write also clears the pending timer interrupt.

## 4.2 PLIC and external interrupts

QEMU `virt` PLIC at `0x0C000000`. Register math (all relative to base):

| Block | Formula |
|---|---|
| Source priority | `+ 0x000000 + 4*source` (0 = disabled) |
| Pending bitmap | `+ 0x001000` |
| Enable bits | `+ 0x002000 + 0x80*context` |
| Threshold | `+ 0x200000 + 0x1000*context` |
| Claim/Complete | `+ 0x200004 + 0x1000*context` |

**Context = `2*hartid + (0 for M-mode, 1 for S-mode)`.** So hart 0's S-mode context is **1** (enable
at `+0x2080`, threshold at `+0x201000`, claim at `+0x201004`). This is the #1 PLIC bug: each hart owns
*two* contexts, so the per-hart stride is twice the per-context stride.

Init: set each source's priority >= 1, set your S-context's enable bit for that source, set the
threshold to 0 (it masks priorities <= threshold). QEMU `virt` source IDs: UART0 = **10**, virtio-mmio
slot 0 = **1**.

The claim/complete handshake, on an S-external interrupt (`scause == 0x8000000000000009`):

```c
int irq = *(uint32*)(PLIC + 0x201004);   // CLAIM: returns highest-pending source, clears its pending
if      (irq == 10) uart_intr();
else if (irq == 1)  virtio_intr();
if (irq) *(uint32*)(PLIC + 0x201004) = irq;   // COMPLETE: write the same id back to re-arm
```

You **must** complete (write the same IRQ back) or the source never re-arms - one interrupt then
silence.

**AIA note:** the modern alternative is the Advanced Interrupt Architecture (IMSIC + APLIC,
message-signaled interrupts), required by the Server SoC spec and used for PCIe/virtualization. QEMU
`virt` can enable it (`-machine virt,aia=...`), but the PLIC path above is the default and what you
implement first.

# Part 5 - Virtual memory (Sv39)

Sv39 gives each process a private 39-bit address space via a three-level page table.

## 5.1 The format

`satp` (CSR `0x180`): bits `[63:60]` MODE (`0`=Bare, `8`=Sv39, `9`=Sv48, `10`=Sv57), `[59:44]` ASID,
`[43:0]` root-table physical page number. `MAKE_SATP(root) = (8L<<60) | (root>>12)`.

**Writing `satp` with an unsupported MODE is silently ignored** - the write is dropped, no trap, and
`satp` keeps its old value (Bare at boot). So a kernel that assumes Sv48/Sv57 on Sv39-only hardware
fails *silently*: paging never turns on (or stays in the prior mode) and you chase phantom faults.
**Read `satp` back after writing it and confirm the MODE field stuck before relying on translation.**

Virtual address split:
```
 38      30 29      21 20      12 11        0
| VPN[2]   | VPN[1]   | VPN[0]   |  offset   |     bits 63:39 must equal bit 38 (else page fault)
   level 2    level 1    level 0    12 bits
```

PTE (64-bit): `V`(0) `R`(1) `W`(2) `X`(3) `U`(4) `G`(5) `A`(6) `D`(7) `RSW`(8-9) `PPN`(10-53)
reserved(54-60) `PBMT`(61-62, Svpbmt) `N`(63, Svnapot). Pack/unpack:
`PA2PTE(pa) = (pa>>12)<<10`, `PTE2PA(pte) = (pte>>10)<<12`, `PX(level,va) = (va>>(12+9*level)) & 0x1FF`.

Leaf vs pointer: `R=W=X=0` => pointer to next level; any of R/W/X set => leaf. **`W=1,R=0` is reserved
and faults.** A leaf at level 1 = a 2 MiB megapage, at level 2 = a 1 GiB gigapage (each needs natural
alignment of the low PPN bits).

## 5.2 Turning paging on

```c
void paging_on(pagetable_t kpt) {
    sfence_vma();                       // (1) drain prior page-table writes
    w_satp(MAKE_SATP((uint64)kpt));     // (2) activate Sv39
    sfence_vma();                       // (3) flush stale/cached (incl. V=0) entries
}
```

**Critical:** the code executing the `csrw satp` must remain mapped at the same VA in the new table on
the very next instruction fetch. Either **identity-map** the kernel (virt == phys, the simplest choice
and what most kernels do) or run the switch from a trampoline mapped at the same VA in both tables. If
the next-instruction VA is not in the new table, you fault immediately and your trap vector is also
unreachable - a silent boot hang.

`walk` (allocate intermediate tables) and `mappages`:

```c
pte_t *walk(pagetable_t pt, uint64 va, int alloc) {
    for (int level = 2; level > 0; level--) {
        pte_t *pte = &pt[PX(level, va)];
        if (*pte & PTE_V) pt = (pagetable_t)PTE2PA(*pte);
        else {
            if (!alloc || (pt = kalloc()) == 0) return 0;
            memset(pt, 0, PGSIZE);
            *pte = PA2PTE(pt) | PTE_V;        // pointer entry: V only, no R/W/X
        }
    }
    return &pt[PX(0, va)];
}
// mappages: for each page, *walk(pt,va,1) = PA2PTE(pa) | perm | PTE_V;
```

**Permissions / W^X discipline:** kernel text and the trampoline = `R|X` (never W); kernel data, all
RAM, and all device MMIO = `R|W` (never X). No page is ever both W and X. Map MMIO as R/W
no-execute.

## 5.3 The A/D bit portability trap

There are two hardware schemes for the Accessed/Dirty bits, and they differ in a way that produces the
infamous "works on QEMU, faults on real hardware" bug:

- **Hardware update (Svadu, and QEMU `virt`'s default):** hardware sets A on access and D on store. A
  kernel that ignores A/D works.
- **Svade (common on real silicon):** an access with A=0, or a store with D=0, raises a **page fault**;
  software must set the bits and retry.

**Portable strategy: if you do not implement swapping, pre-set `A=1` and `D=1` on every leaf PTE at map
time.** This is correct under both schemes. (xv6 leaves A/D to hardware, which is the non-portable
choice - fine on QEMU, a fault loop on Svade hardware.)

## 5.4 TLB coherence

`sfence.vma rs1, rs2`:

| rs1 | rs2 | Flushes |
|---|---|---|
| x0 | x0 | everything (the always-safe over-fence) |
| x0 | asid | all non-global entries for that ASID |
| vaddr | x0 | leaf entries for that VA, all ASIDs |
| vaddr | asid | leaf entries for that VA and ASID |

Rules: you **must** `sfence.vma` after editing any PTE (including making an invalid PTE valid - invalid
entries may be cached). Writing `satp` alone does **not** flush. For non-leaf edits use `rs1=x0`.
"Always legal to over-fence" - a simple kernel can always use `zero,zero`. A local `sfence.vma` only
flushes the issuing hart; other harts need a **TLB shootdown** (IPI + remote `sfence.vma`, or the SBI
RFENCE call). ASIDs let you skip full flushes on context switch (xv6 uses ASID 0 + full flush for
simplicity); on ASID reuse, write `satp` with the recycled ASID first, then `sfence.vma x0, asid`.

`fence.i` (Zifencei) is a *separate* barrier for instruction-memory coherence after you write code
(JIT, loader, trampoline patching). It is **local-hart only** - other harts need their own `fence.i`
(via IPI or SBI `remote_fence_i`).

## 5.5 PMP vs page faults; the sign-extension rule

- A fault on a *physical* access blocked by OpenSBI's PMP is an **access fault** (cause 1/5/7), not a
  page fault (12/13/15). `sstatus.SUM`/`MXR` only affect PTE permission checks - they never rescue a
  PMP fault. A paging-off kernel can still touch MMIO because OpenSBI's PMP grants S/U access to that
  physical region.
- `SUM=1` lets S-mode read/write `U=1` pages (for `copyin`/`copyout`); S-mode can **never execute** a
  `U=1` page regardless of SUM. `MXR=1` lets loads read execute-only pages.
- **Sv39 sign-extension:** a VA's bits `[63:39]` must all equal bit 38, or it faults. So valid VAs are
  a low band and a high (sign-extended) band, with a non-canonical hole between. (xv6 caps `MAXVA =
  1<<38` to stay in the low band.)

## 5.6 COW, demand paging, mmap - one mechanism

All three reuse the same page-fault loop: **fault -> allocate/copy a page -> fix the PTE (set PPN +
permissions, clear any software COW marker in the RSW bits) -> `sfence.vma` -> retry the faulting
instruction.** Copy-on-write fork maps parent and child at the same physical pages read-only with a
COW marker in RSW; the first write faults, the handler copies the page, restores `PTE_W`, and retries.
Lazy allocation and `mmap` install no PTE until first touch; the fault handler allocates and maps with
the region's permissions.

# Part 6 - Atomics and the memory model

RISC-V is weakly ordered (RVWMO): a hart sees its own accesses in order, but other harts may observe
them reordered. Cross-hart and device ordering needs explicit synchronization.

## 6.1 Atomics (the A extension)

- **LR/SC:** `lr.w`/`lr.d` register a reservation; `sc.w`/`sc.d` store only if it holds (writes 0 to rd
  on success, nonzero on failure). For the architectural forward-progress guarantee the loop must be
  **constrained**: <= 16 instructions, naturally aligned operands, and **no loads, stores, backward
  branches, JALR, FENCE, or SYSTEM between the LR and the SC**. A function call or a load in that
  window voids the guarantee and can livelock.
- **AMOs:** `amoswap/amoadd/amoand/amoor/amoxor/amomax/amomin[u].w/.d` - atomic read-modify-write,
  usable on memory-mapped device registers.
- **Ordering bits `.aq`/`.rl`:** `.aq` (acquire) = no later access reorders before; `.rl` (release) =
  no earlier access reorders after. These map to C11/Rust acquire/release.

## 6.2 Spinlocks and the interrupt discipline

```asm
acquire:  amoswap.w.aq t1, t0, (a0)   # swap in 1; t1 = old
          bnez t1, acquire             # retry if it was held
release:  amoswap.w.rl x0, x0, (a0)   # store 0 (release)
```

A C kernel gets this from `__atomic_exchange_n(.., __ATOMIC_ACQUIRE)` and
`__atomic_store_n(.., __ATOMIC_RELEASE)`. **Disable interrupts while holding a spinlock** that might
also be taken from an interrupt handler on the same hart - otherwise the handler spins forever on a
lock the interrupted code holds (self-deadlock). Use a nestable disable (xv6's `push_off`/`pop_off`):
the first acquire disables interrupts and remembers the prior state; the last release restores it.

## 6.3 Fences and MMIO ordering

`fence pred, succ` with bits I (device input), O (device output), R (reads), W (writes):

- `fence rw, rw` - full memory barrier for normal RAM.
- `fence iorw, iorw` - also orders device I/O. The always-safe over-fence for MMIO.

**The device-notify pattern** (required, easy to omit): fill a descriptor/buffer in RAM, **fence**,
then write the device's doorbell register. Without the fence, RVWMO lets the doorbell store be observed
before the buffer fill and the device reads garbage. This is mandatory for virtio (Part 7.4).

# Part 7 - Devices

## 7.1 Device tree (FDT) parsing

RISC-V platform devices have no enumerable bus, so firmware passes a Flattened Device Tree describing
them; you discover hardware from it rather than hardcoding. **Everything in the blob is big-endian.**

Header (40 bytes, ten u32 BE fields): `magic` (`0xd00dfeed`), `totalsize`, `off_dt_struct`,
`off_dt_strings`, `off_mem_rsvmap`, `version` (17), `last_comp_version` (16), `boot_cpuid_phys`,
`size_dt_strings`, `size_dt_struct`.

Struct-block tokens: `FDT_BEGIN_NODE`=1 (then null-terminated name, padded to 4 bytes),
`FDT_END_NODE`=2, `FDT_PROP`=3 (then `{u32 len; u32 nameoff}`, value, padded to 4), `FDT_NOP`=4,
`FDT_END`=9. Property names live in the strings block, indexed by `nameoff`. Re-align to 4 bytes after
every name and every value, or the token stream desyncs.

Properties you parse:

- **`reg`** = (address, size) pairs, decoded with the **parent's** `#address-cells`/`#size-cells`
  (**not the node's own**). Defaults when absent: address-cells **2**, size-cells **1** (note: size
  default is 1, not 2). QEMU `virt` sets root to 2/2, so a `reg` there is a 64-bit base + 64-bit size;
  but a parser that hardcodes 2/2 breaks on `/cpus` (1/0) and on trees using defaults.
- **`compatible`** = most-specific-first driver-match strings.
- **`status`** = "okay"/"disabled"/"reserved"/"fail"; skip non-okay nodes.
- **`/chosen` `stdout-path`** = the console; may be an `/aliases` name and may carry a `:baud` suffix
  (strip at `:`, resolve via `/aliases` if not absolute).
- **`/memory`** (`device_type="memory"`) = RAM base/size.
- **`/cpus`** children (`device_type="cpu"`, skip disabled) = hart count; `/cpus/timebase-frequency` =
  the `time` tick rate (10 MHz on QEMU `virt`).

The pattern: validate magic -> find the console via `/chosen` -> match `compatible` against a driver
registry -> read `reg` for the MMIO base -> instantiate. Discover RAM, harts, and the timer the same
way. This lets one image boot many machines.

## 7.2 MMIO fundamentals

Device registers have side effects, so accesses must be **volatile** (never optimized, reordered, or
coalesced away) - in C, `volatile` pointers; in Rust, `read_volatile`/`write_volatile`; in Go, force
the access so the compiler cannot hoist it out of a poll loop. Device pages are non-cacheable,
no-execute, and need a `fence` when ordering matters across the device boundary (6.3). Use the device's
natural access width and natural alignment.

## 7.3 NS16550A UART

QEMU `virt` at `0x10000000`, `compatible="ns16550a"`, IRQ 10. Byte-wide registers at `base + offset`:

| Off | Read | Write | DLAB=1 |
|---|---|---|---|
| 0 | RBR (rx) | THR (tx) | DLL (divisor low) |
| 1 | IER | IER | DLM (divisor high) |
| 2 | IIR | FCR | |
| 3 | LCR | LCR | |
| 5 | LSR | | |

Bits: `LSR.DR`=`0x01` (rx ready), `LSR.THRE`=`0x20` (tx ready); `IER` rx=`0x01` tx=`0x02`; `FCR` enable=`0x01`
clear=`0x06`; `LCR` 8N1=`0x03`, DLAB=`0x80`.

```c
WriteReg(IER, 0x00);                 // disable interrupts
WriteReg(LCR, 0x80);                 // DLAB on
WriteReg(0, 0x03); WriteReg(1, 0x00);// divisor (= clock/(16*baud); xv6 uses 3)
WriteReg(LCR, 0x03);                 // 8N1, DLAB off  <- must clear DLAB or RBR/THR stay aliased
WriteReg(FCR, 0x07);                 // enable + clear FIFOs
WriteReg(IER, 0x03);                 // enable rx+tx interrupts (omit for pure polling)
// polled TX: while (!(ReadReg(LSR)&0x20)) ; WriteReg(THR, c);
// polled RX: if (ReadReg(LSR)&0x01) c = ReadReg(RBR);
```

Forgetting to clear DLAB (step 4) leaves offsets 0/1 aliased to the divisor and the UART appears dead.
The interrupt path reads IIR to acknowledge, drains the RX FIFO fully (one IRQ can cover several
bytes), and ties into the PLIC claim/complete handshake (4.2).

## 7.4 virtio-mmio (modern, version 2)

For a block device (disk) or NIC. QEMU `virt` slot 0 at `0x10001000`; all registers are 32-bit.

| Off | Register | | Off | Register |
|---|---|---|---|---|
| 0x000 | MagicValue (`0x74726976`) | | 0x044 | QueueReady |
| 0x004 | Version (2) | | 0x050 | QueueNotify |
| 0x008 | DeviceID (2=blk,1=net) | | 0x060 | InterruptStatus |
| 0x010/0x014 | DeviceFeatures/Sel | | 0x064 | InterruptACK |
| 0x020/0x024 | DriverFeatures/Sel | | 0x070 | Status |
| 0x030 | QueueSel | | 0x080/4 | QueueDescLow/High |
| 0x034 | QueueNumMax | | 0x090/4 | QueueDriverLow/High (avail) |
| 0x038 | QueueNum | | 0x0a0/4 | QueueDeviceLow/High (used) |

Status bits: ACKNOWLEDGE=1, DRIVER=2, DRIVER_OK=4, FEATURES_OK=8. Init handshake (order matters):

```
check Magic==0x74726976, Version==2, DeviceID==2
Status = 0                         # reset
Status |= ACKNOWLEDGE; |= DRIVER
read DeviceFeatures; write DriverFeatures (clear bits you won't use)
Status |= FEATURES_OK; re-read Status -> if FEATURES_OK didn't stick, abort
QueueSel=0; check QueueNumMax>=N; QueueNum=N
allocate desc/avail/used (page-aligned, zeroed); write their physical addrs to
   QueueDesc/Driver/Device Low/High; QueueReady=1
Status |= DRIVER_OK
```

The split virtqueue: `virtq_desc{ u64 addr; u32 len; u16 flags; u16 next; }` (flags NEXT=1, WRITE=2,
INDIRECT=4); `virtq_avail{ u16 flags; u16 idx; u16 ring[N]; }` (driver-owned); `virtq_used{ u16 flags;
u16 idx; struct{u32 id; u32 len;} ring[N]; }` (device-owned). A block request is a 3-descriptor chain:
header `{u32 type (0=read,1=write); u32 reserved; u64 sector}` (device reads), the data buffer (mark
`WRITE` on a disk *read* so the device writes it), and a 1-byte status (device writes 0=OK).

**Publish + notify, with the mandatory fences:**

```c
avail->ring[avail->idx % N] = head;
fence();                              // ring write before idx bump
avail->idx += 1;                      // free-running counter, NOT % N
fence();                              // idx bump before the doorbell
*(uint32*)(VIRTIO + 0x050) = 0;       // QueueNotify = queue index (kick)
```

On completion (the device raises its PLIC IRQ): write `InterruptStatus & 3` back to `InterruptACK`
(or the device never interrupts again), then walk the used ring (`while (used_idx != used->idx)`),
read each completed chain's `id` and status, wake the waiter, advance `used_idx`. Legacy (version 1)
virtio uses a single `QueuePFN` region instead of the three address registers and has no FEATURES_OK
step - target v2.

# Part 8 - Processes and concurrency

## 8.1 The process abstraction

A process is: its own Sv39 page table (the `satp` value when it runs in U), a trapframe (3.3), a
kernel stack (with an unmapped guard page below it), a saved kernel-thread context, and a state. Map
the trampoline and trapframe into every user page table at fixed high VAs.

## 8.2 The system-call path

User sets the call number in `a7` and args in `a0`-`a5`, executes `ecall` (-> cause 8). The trampoline
saves the user GPRs and switches to the kernel table (3.3); `usertrap` advances `sepc` past the ecall,
dispatches through a syscall table, puts the result in `a0`, and `userret` switches back and `sret`s.
**Never dereference a user pointer directly:** either software-walk the user page table checking
`PTE_U`/`PTE_W` (xv6's `copyin`/`copyout`), or set `sstatus.SUM` briefly with a fault fixup (Linux).
The hardware guarantees S-mode cannot *execute* a user page regardless of SUM.

## 8.3 Trap vs context switch

These are different mechanisms:

| | Trap (user<->kernel) | Context switch (kernel thread<->thread) |
|---|---|---|
| Saves | all 31 GPRs to the trapframe | only callee-saved `ra,sp,s0-s11` to a `context` struct |
| Privilege | yes (U<->S) | no (stays in S) |
| `satp` | switches (via trampoline) | no change |
| Resume PC | `sepc` | implicit, in `ra` |

`swtch(old, new)` saves `ra,sp,s0-s11` into `old` and loads them from `new`, then `ret` - control
returns into a different thread. A reschedule is typically two `swtch` calls through a per-CPU
scheduler thread. Hold the process lock across the switch and release it in the destination thread, so
no other hart runs the process until it is fully off its stack. The timer interrupt (4.1) drives
preemption by calling `yield()` from the trap handler.

## 8.4 SMP bringup

`mhartid` is M-mode-only, so stash each CPU's id in `tp` at entry and read it via `tp` thereafter.
Bring up secondaries on demand with **SBI HSM `hart_start`** (2.2) - each enters in S-mode with
`satp=0`, `SIE=0`, `a0`=hartid, `a1`=opaque. Cross-hart coordination: **IPIs** via `sip.SSIP`
(or SBI `send_ipi`), and **remote TLB shootdown** via SBI RFENCE (`remote_sfence_vma`), since a local
`sfence.vma` only flushes the issuing hart.

## 8.5 Synchronization in the kernel

Use spinlocks (6.2) for short critical sections with the interrupt discipline, and sleeplocks (built on
spinlocks + sleep/wakeup) for long, blocking sections. A managed-runtime kernel may instead use the
runtime's mutexes/channels for kernel logic - those still bottom out on `lr/sc`/`amoswap` and the same
acquire/release ordering.

# Part 9 - Modern features (optional)

These are real but not needed for a first working kernel. Add them when the base is solid.

- **Virtualization (H extension):** adds HS/VS/VU modes and two-stage translation. Stable since ~2021
  (KVM/QEMU implemented it early), formally ratified within Privileged Architecture v1.13 (~2025).
  **KVM/RISC-V** has been mainline since Linux 5.16. Needs the **AIA** (IMSIC/APLIC) for efficient
  virtualized interrupts and the **RISC-V IOMMU** (mainline Linux 6.13) for device passthrough.
- **Security:** **Smepmp/ePMP** (ratified 2021; M-mode lockdown via `mseccfg` MML/MMWP/RLB), **pointer
  masking** (Smmpm/Smnpm/Ssnpm, ratified 2024; Linux 6.13), **CFI** (Zicfilp landing pads + Zicfiss
  shadow stack, ratified). **CHERI** for RISC-V (the "Y" extension) was still **draft** as of mid-2026.
  **Confidential computing** (CoVE / AP-TEE + the Smmtt supervisor-domains hardware) was still draft.
  **Keystone** enclaves (PMP-based, M-mode security monitor) are a research framework.
- **Real-time:** **PREEMPT_RT** is mainline (Linux 6.12, includes RISC-V); lowest-latency interrupt
  work centers on the **CLIC** (still draft), distinct from the scale-oriented AIA.
- **W^X / safe-language angle:** enforce W^X via PTE bits (5.2). Memory-safe-language kernels (the Go
  Biscuit research kernel, on x86-64) found a large fraction of memory-corruption CVEs become panics
  rather than exploits, at a measured cost of up to ~13% kernel CPU.

# Part 10 - Ecosystem and targeting context

Where a from-scratch kernel sits, and what shapes its choices (production-weighted; mid-2026):

- **Linux dominates RV64.** Mainline `arch/riscv` is feature-complete (SMP, KVM, vector since 6.5, AIA
  since 6.10, IOMMU 6.13). **Debian** carries riscv64 as an official release architecture (Debian 13,
  2025-08-09); **Ubuntu** moved its baseline to RVA23 in 25.10; **Fedora** is still an alternative
  arch. **Android** boots in AOSP but is pre-production (no GKI). The userspace baseline is
  `rv64gc`/lp64d; Rust is Tier 2.
- **Other production OSes:** **FreeBSD** (Tier 2) and **OpenBSD** (release arch since 7.0) are
  release-quality; **NetBSD** ships RISC-V in the imminent 11.0; **seL4** is the only formally verified
  option (uniprocessor, verified on the SiFive HiFive Unleashed / U540, first 64-bit arch verified to
  binary); **Zephyr** and **FreeRTOS** are the embedded mainstays. **Fuchsia, Redox, Xen** are
  emerging/QEMU-stage. **Windows** has no port.
- **Standards:** application **profiles** (RVA20/22/23) fix the ISA baseline; the **Server SoC** (2025)
  and **BRS** (2025) specs define the firmware/boot contract for servers (UEFI+ACPI), while embedded
  uses devicetree (EBBR). SBI is the firmware ABI everywhere.
- **Hardware reality:** shipping SBCs are RVA20/RVA22-class - StarFive JH7110 (VisionFive 2, no
  vector), SpacemiT K1/X60 (RVV 1.0, VLEN=256), the Sophgo SG2042 64-core workstation, SiFive P550
  boards. The **RVV vector fragmentation** (T-Head's non-standard 0.7.1 vs ratified 1.0) fractured
  toolchains for years and is now mostly healed in new silicon. **RVA23 silicon barely existed through
  2025** (first consumer SoC taped out Dec 2025), so distros target a baseline current boards don't
  meet. Application-core performance still trails comparable Arm/x86. Industry consolidation is real:
  **Qualcomm acquired Ventana Micro (Dec 2025).**

Implication for your kernel: target `rv64gc` and discover everything else from the device tree; depend
on SBI (>= v2.0, with HSM) for firmware services; assume devicetree discovery (not ACPI) on the boards
you can actually buy; and treat vector, H, and AIA as feature-detected extras, not a baseline.

# Appendix C - Trap cause codes

`scause` bit 63 = 1 -> interrupt, 0 -> exception.

Exceptions: 0 instr-misaligned · 1 instr-access-fault · 2 illegal-instruction · 3 breakpoint ·
4 load-misaligned · 5 load-access-fault · 6 store/AMO-misaligned · 7 store/AMO-access-fault ·
8 ecall-from-U · 9 ecall-from-S · 11 ecall-from-M · 12 instr-page-fault · 13 load-page-fault ·
15 store/AMO-page-fault · (16 double-trap · 18 software-check · 19 hardware-error, newer spec).

Interrupts (bit 63 set): 1 S-software · 5 S-timer · 9 S-external · (3/7/11 = M-software/timer/external;
13 = counter-overflow, Sscofpmf). The cause number equals the bit position in `sie`/`sip`.

# Appendix F - Gotchas that brick the kernel (master list)

1. Linking at `0x80000000` under `-bios default` -> overlaps OpenSBI. Link at `0x80200000`.
2. Passing a flat `objcopy` binary to `-kernel` (silent no-boot). Pass the ELF.
3. `-mcmodel=medlow` -> cannot address `0x80200000`. Use `medany`.
4. Setting `gp` with relaxation on -> `mv gp, gp`, gp stays garbage. Use `.option norelax` / `--no-relax`.
5. Not clearing BSS (firmware doesn't). Globals hold garbage.
6. Clobbering `a0`/`a1` (hartid/DTB) before saving them across the BSS memset.
7. Unaligned `sp` (must be 16-byte aligned at any call).
8. `sstatus.FS` left Off -> first lp64d FP instruction traps. Set FS=Initial in `_start`.
9. Switching `satp` from non-identity-mapped (or non-trampoline) code -> immediate fetch fault.
10. Relying on `csrw satp` to flush the TLB - it doesn't. Bracket with `sfence.vma`.
11. Writing `satp` with an unsupported MODE is silently ignored (the write is dropped) -> paging never
    turns on / stays in the old mode, no trap. Read `satp` back to confirm the MODE stuck.
12. Missing `sfence.vma` after making a PTE valid (invalid entries may be cached).
13. Assuming hardware sets A/D - faults on Svade hardware. Pre-set A=D=1 at map time.
14. Non-canonical Sv39 VA (bits 63:39 must equal bit 38).
15. Re-enabling interrupts in a trap handler before spilling `sepc`/`scause` (one-deep CSR stack).
16. Not advancing `sepc` past `ecall` (+4) -> infinite trap loop.
17. PLIC context math: hart `h` S-mode is context `2h+1`, not `h`. Wrong stride -> handler never fires.
18. Not writing PLIC complete (same IRQ back) -> source never re-arms.
19. Missing the fences before virtio `avail.idx++` and before the QueueNotify doorbell.
20. Not ACKing virtio `InterruptStatus` -> no further device interrupts.
21. virtio descriptor `WRITE` flag backwards (device-writes-buffer is for disk *reads*).
22. FDT read as little-endian (it's big-endian) or `reg` decoded with the wrong (non-parent) cells.
23. Leaving the UART DLAB bit set -> RBR/THR stay aliased to the divisor; UART looks dead.
24. `fence.i` assumed global - other harts don't see patched code (works on QEMU, fails on hardware).
25. Spinlock held with interrupts enabled while the same lock is taken from an interrupt handler ->
    self-deadlock.

# Appendix G - Recommended bringup order

1. **Boot to S-mode.** Linker script + `_start` + the QEMU command line. Print one byte via SBI
   `console_putchar` to prove you reached `kmain` with `a0`/`a1` intact.
2. **SBI console + early print.** A formatted print over SBI (or DBCN). Now you can debug by printf.
3. **Parse the device tree** from `a1`: RAM base/size, hart count, the UART base, the timebase. Switch
   the console to the discovered NS16550 (polled).
4. **Trap handler.** Install a Direct-mode `kernelvec`; decode `scause`; handle nothing yet but panic
   cleanly with `scause`/`sepc`/`stval` on an unexpected trap.
5. **Physical page allocator** (a free list over the RAM the DTB reported, minus the kernel image).
6. **Sv39 paging.** Build a kernel table (identity-map the kernel R/X and R/W, and the MMIO R/W),
   activate it (the `sfence`-bracketed `csrw satp`), confirm you still run. Pre-set A/D.
7. **Timer.** Detect Sstc; arm via `stimecmp` or SBI `set_timer`; take periodic S-timer interrupts.
8. **PLIC + interrupt-driven UART** (claim/complete). Now you have asynchronous input.
9. **Processes:** trampoline + trapframe, the syscall path, a context switch, a scheduler. Run a
   U-mode program behind `ecall`.
10. **virtio-blk + a file system**, then fork/exec/wait, then COW/demand-paging, then SMP (HSM
    `hart_start` + IPIs + RFENCE shootdown).

Each step is testable in QEMU before the next. Keep the "correct on QEMU, broken on hardware" traps
(Appendix F: A/D bits, `fence.i`, MMIO fences, missing `sfence.vma`) in mind from the start - they do
not show up until you run on real silicon.
