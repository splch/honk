# honk v1 — Release Requirements

**An everyday operating system for the average person.**

Status: **DRAFT / proposed.** This document defines what honk must do, and how
well, before it can be called a *v1* that an ordinary person can use as their
normal computer. It turns the gap analysis in [`docs/OS.md`](OS.md) and the
project audit into concrete, testable requirements. It is forward-looking: most
of what follows is **not yet built** (today honk is M0–M12 complete with the
M13 hypervisor mechanisms proven — see [`docs/STATUS.md`](STATUS.md)).

> Companion docs: [`HONK.md`](HONK.md) (the design and the OS-concept→Go-primitive
> mapping), [`STATUS.md`](STATUS.md) (what is actually built today), [`RV64.md`](RV64.md)
> (the hardware reference), [`OS.md`](OS.md) (what people actually do with computers),
> [`GO.md`](GO.md) (the coding bar).

---

## 0. The v1 thesis (the load-bearing decision)

An "everyday OS for the average person" is, in 2026 terms, **a personal computer
you live in**: it boots on real hardware, logs you in, connects to wifi, and runs
a **web browser**, video, messaging, email, a media player, an office/document
app, and a handful of installed apps — reliably, securely, on battery, without a
terminal. (See `OS.md`: daily life is browsing, video ~3h/day, social, messaging,
and the new generative-AI workload, almost all of it mediated through a browser
and ~a dozen apps.)

honk cannot meet that by writing native apps. There is no pure-Go web browser, no
app ecosystem, and wazero is interpreter-only on riscv64. The long tail of
everyday software exists only for Linux. honk already has the one mechanism that
turns this from impossible to tractable: **it is a Type-1 hypervisor.**

> **v1 architectural decision (locked):** honk ships as a **memory-safe host that
> owns the hardware, the user's identity and data, security, the network, and the
> desktop compositor — and runs everyday applications inside Linux guest VMs whose
> windows are composited into honk's desktop.** This is the proven
> ChromeOS-Crostini / Qubes / Firecracker-desktop model. honk's small,
> memory-safe TCB and its existing immutability/capability discipline are exactly
> what make this *safe*; the H-extension is how it is *possible*.

Consequences that shape every requirement below:

- The **browser and the app long tail run in Linux guests** (REQ-RUNTIME, REQ-APPS,
  REQ-BROWSER). honk does **not** reimplement them.
- honk's own pure-Go code is the **host**: boot, drivers, power, storage, network,
  identity, crypto, compositor, settings, updater, and the VMM. This is where the
  small-TCB, memory-safe thesis pays off.
- honk's three isolation tiers stay intact: trusted host **goroutines**, optional
  **WASM** apps, and **VM** apps. The everyday tier is the VM.

Everything in this document is a requirement on **the host (honk) and the guest
*integration*** — never on the guest applications themselves.

## 1. Definition of done — the "average person" acceptance scenarios

v1 is "perfectly viable" only if a non-technical person can do **all** of the
following on supported hardware, with no terminal and no documentation, each as a
pass/fail acceptance scenario (the full gate is §7):

1. **Set it up.** Power on a new machine, follow a graphical first-run wizard
   (language, wifi, account, timezone), and reach a desktop in < 10 minutes.
2. **Get online.** Join a wifi network from a menu; the connection persists across
   reboot.
3. **Browse the web.** Open the browser, sign in to a webmail and a social/video
   site, watch a 1080p video with smooth playback and audio, and use a web app
   (docs, maps) with a working keyboard, mouse/trackpad, copy/paste, and file
   upload/download.
4. **Do a video call.** Join a standard web video call with working camera, mic,
   speaker, and screen share.
5. **Handle files & media.** Plug in a USB drive, browse it in a graphical file
   manager, copy photos to the machine, view them, and safely eject.
6. **Install and run an app** from a graphical catalog, launch it from the app
   menu, and have it survive a reboot.
7. **Stay current and recover.** Receive and apply an update in the background;
   reboot into the new version; and, if an update is bad, automatically roll back
   to a working system with no data loss.
8. **Be secure by default.** Data at rest is encrypted; the screen locks and
   requires the account password/PIN; a second account cannot read the first's
   files; the system survives a malicious web page or app without compromise.
9. **Live on battery.** Suspend on lid-close, resume in < 2 s, show accurate
   battery level, and idle for hours without draining (no busy-spin).
10. **Be usable by anyone.** Full keyboard with shift/symbols/layouts, adjustable
    text size and contrast, and a screen reader for the core shell.

If any of 1–10 fails, honk is **not** v1.

## 2. Non-goals for v1 (explicitly out of scope)

- **Being a UNIX / POSIX host for native server software.** Native everyday/server
  apps are not a goal; that software runs in guests.
- **A pure-Go web browser or office suite.** Delivered via guests.
- **Phone/tablet form factor**, cellular/5G, touch-first UI. (Desktop/laptop-class
  first; touch is post-v1.)
- **Gaming with 3D acceleration / GPU passthrough**, professional A/V production,
  and other heavy-GPU workloads. (Software/virgl rendering only in v1.)
- **Enterprise fleet management, domain join, MDM.** Post-v1.
- **Broad hardware support.** v1 targets a **small reference hardware set** (§3),
  not "runs on anything."
- **Backwards-compatible native ABI / third-party native kernel modules.** honk's
  drivers are first-party and in-tree.

## 3. Reference hardware (the v1 support set)

v1 must run, fully, on a **single, named reference machine** plus QEMU; breadth
comes later.

| ID | Requirement | Priority |
|---|---|---|
| REQ-HW-1 | Define one reference laptop/desktop-class RISC-V machine (e.g. an RVA22/RVA23-class board with the H-extension, an attached display, USB, NVMe/SD, and a supported wifi NIC). All v1 acceptance runs on it. | **P0** |
| REQ-HW-2 | Continue to boot and pass the full host test/smoke suite on QEMU `virt` for development and CI. | **P0** |
| REQ-HW-3 | Hardware requires the **H-extension** (mandatory: the app runtime is VMs) and **RVA22+** baseline; feature-detect everything else from the device tree. | **P0** |
| REQ-HW-4 | A documented, reproducible way to produce installation media for the reference machine. | **P0** |
| REQ-HW-5 | A second reference device (different NIC/GPU/storage) to prove the DTB-driven discovery generalizes. | P1 |

Today: QEMU `virt` only, with QEMU-coherent-DMA assumptions and no real-silicon
A/D-bit/fence/cache handling (`RV64.md` App. F). REQ-HW-1/3 are net-new.

## 6. Architecture deltas required (summary)

The new subsystems v1 introduces on top of today's host:

- **VMM → app runtime:** multi-guest scheduler, hart time-sharing, PLIC/AIA, a full
  guest DT, and the virtio backend suite (fs/blk/net/gpu/input/console/sound/cam).
- **Compositor + desktop shell:** window manager, damage tracking, guest-window
  composition, launcher/tray/notifications, and the host apps (Settings, Files,
  Setup).
- **Connectivity stack:** DHCP/IPv6, wifi (WPA supplicant equivalent), firewall,
  NAT/bridge to guests, DNS/NTP.
- **Storage stack:** a streaming file store + encryption + removable media +
  virtio-fs backend, with the KV store retained for system state.
- **Identity & security:** accounts/login/lock, FDE, hardware root of trust,
  enforced capability/permission model, permission prompts.
- **Power:** WFI/IPI idle, suspend/resume, battery.
- **Input/i18n/a11y:** full keyboard + layouts + IME, trackpad, localization,
  accessibility.
- **Peripherals:** USB host stack, audio, camera/mic.
- **Lifecycle:** in-system A/B updater with health-gated activation + rollback,
  app catalog, logging/diagnostics.

Each maps to the milestones in §8 and the requirements in §4.

## 7. Release acceptance gate ("perfectly viable" definition)

v1 ships only when **all P0 and P1 requirements are met** and **all ten §1
scenarios pass on the reference hardware (REQ-HW-1)**, executed by a
non-technical tester with no documentation, *and* the engineering gates hold:

- **Scenario gate:** §1.1–§1.10 all pass on the reference board.
- **Resilience gate:** kill power at 100 random instants during heavy use →
  always boots back to a consistent system with no acked-data loss (REQ-NFR-REL-2);
  a deliberately malicious guest and a crashing app never compromise or hang the
  host (REQ-SEC-9, REQ-NFR-REL-1).
- **Update gate:** push a good update (applies + persists) and a deliberately bad
  update (auto-rolls-back, no data loss) (REQ-UPD-1/2).
- **Security gate:** FDE on by default, RNG is real (REQ-SEC-1), firewall
  default-deny, second account cannot read the first's data, screen lock holds,
  `govulncheck` clean, and an external security review of the host TCB is passed.
- **Power gate:** measured idle drain within target; suspend/resume < 2 s; battery
  reporting accurate (REQ-PWR).
- **Quality gate:** `REQ-NFR-QUAL-1` green in CI on both QEMU and the reference board.
- **Accessibility gate:** the core shell passes the stated a11y baseline.

If any gate fails, it is not v1.

## 9. Risks and mitigations

| Risk | Mitigation |
|---|---|
| **Scope:** v1 is a multi-year effort an order of magnitude larger than M0–M12. | Sequence by the milestone chain; keep each milestone shippable/testable; lean hard on reuse (Linux, virtio, stdlib) — write only the host. |
| **The app runtime (M13) is the critical path and the hardest piece.** | It is already de-risked mechanism-by-mechanism (`STATUS.md`); the remaining work is integration against a real kernel, not invention. |
| **Real-silicon traps** (A/D bits, fences, cache, non-coherent DMA) not exercised today. | M14 dedicated to hardware bring-up on REQ-HW-1; keep the QEMU+board dual CI gate. |
| **Performance** of interpreted/virtualized everyday apps. | Everyday apps run as native code *inside* guests (not WASM); target software-rendered 1080p; defer GPU accel to post-v1. |
| **Security surface grows** with guests, networking, accounts. | Keep the host TCB small and memory-safe (REQ-SEC-7); confine every app to a guest/WASM sandbox with enforced grants; external review at M22. |
| **`tamago-go` runtime changes** (WFI/IPI idle, possibly suspend). | Treat as tracked, minimal, well-tested forks gated behind the scheduler/IRQ stress suites; upstream where possible. |
| **Hardware availability** (RVA22/23 laptops are scarce as of 2026, `RV64.md` §10). | Pick the reference board pragmatically; QEMU remains the always-available dev/CI target; broaden hardware post-v1. |

## 10. Traceability — audit findings → requirements

| Audit finding (blocker/gap) | Addressed by |
|---|---|
| No memory isolation / advisory-only capabilities for native code | REQ-SEC-4, REQ-APPS-3 (everyday code runs as confined guests/WASM with enforced grants) |
| No preemption of uncooperative trusted code | REQ-RUNTIME-5/6 (apps are preemptible guests, not host goroutines) |
| Filesystem is not a general filesystem | REQ-FS-1/2/4/5/7 |
| QEMU-only; coherent-DMA assumptions; no power mgmt | REQ-HW-1/3, REQ-PLAT-2/3/4, REQ-PWR-* |
| Insecure RNG | REQ-SEC-1 |
| No users/login/permissions | REQ-SEC-2/5, REQ-APPS-3 |
| No in-system update/recovery runtime | REQ-UPD-1..4 |
| Networking appliance-fixed (static IP, HTTP-only, polled) | REQ-NET-1..5, REQ-PLAT-4 |
| GUI is a 2-widget demo; no shift key | REQ-GUI-*, REQ-INPUT-1/2 |
| WASM interpreter-only; no everyday app path | REQ-RUNTIME-*, REQ-BROWSER-1, REQ-APPS-* |
| Thin operability (no slog/pprof/metrics/logging) | REQ-OPS-1..4 |
| Strengths to preserve: memory-safe small TCB, crash-safe KV, signed A/B verity, hypervisor | REQ-SEC-7, REQ-FS-6, REQ-SEC-3, REQ-RUNTIME-* |
